# 📌 创意背景
> 说明这个创意从哪里来，想解决什么真实问题，为什么值得做。

**创意来源，为什么有这个创意，解决什么问题：**

在企业 On-Call（值班）场景里，真正困难的往往不是“没有文档”，而是“告警来了，人在高压状态下找不到正确 SOP（标准作业流程），也不知道 AI 给出的建议能不能信”。新人值班时不知道该搜哪个 SOP；老同学也常常要在告警、文档、历史经验和人工判断之间来回切换。通用大模型虽然能快速回答，但如果没有明确依据，很容易给出看似合理、实际不可执行甚至有风险的建议。

On-Call Copilot 的创意来自这个真实痛点：把分散的 SOP 文档升级成一个可信可观测的值班 Agent（智能体）。用户只需要用自然语言描述故障，Agent 就能自动检索、读取文档、生成带依据的处置建议，并把“读了哪些文件、调用了哪些工具、当前回答依据是什么”展示出来。

**目标用户 / 使用对象：**

- SRE（站点可靠性工程）同学
- 后端值班同学
- 业务系统 Owner（负责人）
- 一线故障响应人员
- 刚接触值班流程、需要快速理解 SOP 的新人

**使用场景：**

针对值班同学，在服务告警、故障响应、应急排查等场景下，为了解决“找不到 SOP、看不懂流程、AI 回答不可信”的问题，提供一个能检索知识、读取 SOP、生成可追溯建议的 Agent 助手。

典型问题包括：

- 服务 OOM（内存溢出）了怎么办？
- 数据库主从延迟超过 30 秒怎么处理？
- P0 故障响应流程是什么？
- 怀疑有人入侵系统怎么排查？
- 推荐结果质量下降如何定位？

**现有方案以及为什么需要新的Agent方案：**

|*现有方案 / 做法*|*优点*|*问题 / 不足*|*新Agent方案可以带来哪些变化*|
|-|-|-|-|
|人工翻 SOP 文档|结果相对可靠，符合团队已有规范|搜索成本高，新人不知道关键词，高压场景下容易漏步骤|用户用口语描述问题，Agent 自动检索并读取相关 SOP|
|问有经验的同学|响应快，上下文丰富|依赖个人经验，占用他人时间，知识难沉淀|把经验入口产品化，降低重复问人|
|通用 AI 问答|表达自然，响应速度快|容易没有来源，可能幻觉，不适合直接指导故障操作|回答基于 SOP，展示读取文件和工具调用过程|
|普通关键词搜索|简单稳定，易实现|对口语化问题、中文短词、英文缩写不友好|关键词检索 + 语义检索 + RRF（倒数排名融合）混合排序，提高召回稳定性|

# 🎯 解决方案
> 说明作品是什么，怎么解决问题，核心功能是什么。

**解决方案简介：**

On-Call Copilot 是一个面向企业值班场景的可信 Agent 助手。它不是普通聊天机器人，而是一条可验证的值班处置链路：先检索，再读 SOP，再基于证据回答，最后把工具调用和依据展示给用户。

一句话说，它把 SOP 从“文档堆”变成“可问、可查、可追溯的值班驾驶舱”。

**核心功能/价值：**

1. **关键词检索**：支持 OOM、CDN、特殊符号、中文短词等查询。
2. **语义检索**：把“服务器挂了”“黑客攻击”等口语问题映射到正式 SOP。
3. **RRF 混合排序**：融合关键词和向量检索结果，提高召回稳定性。
4. **Agent 工具调用**：Agent 自动选择并读取相关 SOP，而不是凭空回答。
5. **流式回答**：用 SSE（服务端事件）逐步输出处理建议，体验更像真实助手。
6. **工具调用可视化**：前端展示“正在读取 sop-xxx”，让用户看到依据链。
7. **防幻觉设计**：通过工具限制、Prompt（提示词）铁律、硬门、引用强制和前端可观测降低编造风险。
8. **Fallback（降级）模式**：LLM（大语言模型）不可用时，仍能通过语义检索和模板回答完成演示。

**用户使用流程：**

1. 用户打开前端页面：`http://127.0.0.1:8000`。
2. 用户选择“关键词搜索”“语义搜索”或“AI Agent 对话”。
3. 用户输入故障问题，例如“服务 OOM 了怎么办？”。
4. 系统先检索 SOP，找到相关文档。
5. Agent 调用 readFile 工具读取 SOP 原文。
6. 前端展示 Agent 正在读取的文件，例如 `sop-001`。
7. Agent 基于文档生成处理建议，并流式输出。
8. 用户根据建议执行排查，同时可以看到建议来源。

**作品亮点：**

1. **可信：回答有依据**。Agent 不是凭模型记忆直接回答，而是先读 SOP，再生成建议。
2. **可观测：过程看得见**。前端展示工具调用、读取文件、相关性分数、流式输出和 `/status` 状态。
3. **工程完整：不是概念 Demo**。项目包含前端、后端、关键词检索、语义检索、Agent Loop（智能体循环）、状态接口和验证用例。

# 🤖 Agent / AI 能力说明
> 本季 Hackathon 重点关注 Agent 和 AI 对工作流、产品形态和个人能力的重塑。请重点说明作品中 AI / Agent 扮演了什么角色。

## 1. 使用了哪些 AI / Agent 能力？

- 大模型理解 / 生成
- Agent 工具调用
- RAG（检索增强生成）/ 知识库检索
- SOP 文档读取
- 流式回答
- 防幻觉 Prompt 约束
- 可观测 Agent 执行过程
- Fallback 降级能力

## 2. Agent 在作品中承担什么角色？

|*Agent 角色*|*是否使用*|*说明*|
|-|-|-|
|*助手*|是|帮助值班同学理解故障问题并生成处理建议|
|*分析师*|是|根据用户问题匹配相关 SOP，提炼关键步骤|
|*执行者*|是|调用 readFile 工具读取 SOP 原文|
|*协作者*|是|用户提出问题，Agent 给出带依据建议，辅助人完成排障|
|*流程编排者*|部分使用|按“检索 → 读文档 → 回答 → 展示依据”的链路组织执行过程|

## 3. 和普通 AI 问答相比，新在哪里？

On-Call Copilot 不是“模型凭经验回答”的普通 AI 问答，而是“Agent 带证据协助处置”的工作流工具。它的新意主要体现在：

- 连接真实 SOP 文档，而不是只依赖模型记忆。
- 能调用 readFile 工具读取文件，而不是只生成文本。
- 能展示读了哪些 SOP，让用户判断可信度。
- 能从输入到输出形成闭环：问题输入 → SOP 检索 → 文档读取 → 建议生成 → 依据展示。
- 面向真实值班流程，而不是泛用聊天。

## 1. 可体验链接
*如果作品可体验，请说明如何使用。*

- 访问链接：
  - GitHub 创意详情页：https://github.com/Yang1999code/on-call-assistant
  - 本地体验地址：`http://127.0.0.1:8000`
- 登录方式：本地运行，无需登录。
- 体验步骤：
  1. 克隆或打开项目代码。
  2. 安装依赖：`pip install -r backend/requirements.txt`。
  3. 启动服务：`python -m uvicorn backend.main:app --host 127.0.0.1 --port 8000`。
  4. 浏览器访问：`http://127.0.0.1:8000`。
  5. 选择 AI Agent 对话 Tab。
  6. 输入推荐问题进行体验。
- 注意事项：
  - 启动时会自动索引 `data/` 目录下的 SOP 文档。
  - 如果配置 LLM API Key，会进入完整 Agent 模式。
  - 如果未配置 LLM API Key，会进入 Fallback 降级模式，仍可完成语义搜索和基础建议生成。

- `服务 OOM 了怎么办？`

- `P0 故障的响应流程是什么？`

# 📍 预期价值
> 说明该创意如果落地，预计能给业务带来什么价值。可以是效率、成本、体验、收入、质量、规模化推广等。

**对个人值班同学：**

- 降低新人值班门槛。
- 减少翻文档和问人的时间。
- 提高故障响应时的确定性。

**对团队：**

- 把分散 SOP 变成统一问答入口。
- 降低重复咨询和知识流失。
- 让故障处理流程更标准、更可追溯。

**对业务：**

- 缩短 MTTR（平均恢复时间）。
- 降低误操作风险。
- 提升服务稳定性和故障响应质量。

**对 Agent 产品形态：**

On-Call Copilot 展示了一种更务实的企业 Agent 路线：不是让 AI 替代人做最终决策，而是让 Agent 在高压工作流中承担“检索、读取、整理、解释、提示风险”的协作角色。

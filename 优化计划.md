## 二、优化项清单

### 🔴 P0 — 必须改（影响面试官对正确性的判断）

#### P0-1: 删除 `VERIFIER_PROMPT`（5 分钟）

**文件**: [prompts.py:26-34](backend/phase3/prompts.py#L26-L34)

**问题**: 定义了但从未调用，留着会让面试官误以为代码在跑双 Agent。GPT 纠错中 40% 打偏的批评都源于这种误解——面试官看代码时发现了 `VERIFIER_PROMPT`，以为这是实际运行的"双 Agent 验证管道"，于是批评"太重"。

**改法**: 直接删除第 26-34 行。README 中关于 "Verifier Agent 预留" 的描述可以保留（设计文档），但代码里不要留未使用的 Prompt。

```python
# 删除这段（prompts.py:26-34）
VERIFIER_PROMPT = """You are a verification agent. ..."""
```

#### P0-2: 补充特殊字符检测（10 分钟）

**文件**: [search_engine.py:45](backend/phase1/search_engine.py#L45)

**问题**: 当前只用 `len(q) < 3` 来判断是否走 LIKE fallback。`&`（len=1）能兜住，但 `A&B`（len=3）会进 FTS5 MATCH 分支，FTS5 的 trigram tokenizer 对 `&` 的行为不可预测。

**改法**: 在 `len(q) < 3` 旁加特殊字符正则检测。

```python
# search_engine.py 顶部加 import
import re

# Line 45 改为:
SPECIAL_CHARS_RE = re.compile(r'[&%_\\/\-]')
if len(q) < 3 or SPECIAL_CHARS_RE.search(q):
    # 短查询或含特殊字符 → LIKE 回退
```

**影响范围**: 不影响现有 20/20 测试结果，增加对 `A&B`、`N/A` 等边界查询的防御。

### 🟡 P1 — 建议改（提升面试体验）

#### P1-1: 提供一键 LLM 体验方案（20 分钟）

**问题**: 当前 Fallback 模式（无 API Key）只是"语义搜索 + 读文档 + 模板回答"。Phase 3 占 40 分，面试官如果只看 Fallback 模式，无法评估 LLM Agent 的真实能力。

**分析**:

题目没有要求提供 API Key，面试官自己也没有义务配 Key。但：
- 如果面试官恰巧有 DeepSeek/OpenAI Key，配上去就是完整 LLM Agent
- 如果面试官没有，Fallback 的"只读文档"演示不了 Agent 的综合推理

**方案 A（推荐）**: 不改代码，在 README 显眼位置写清楚两部分效果差异，并提供**模拟 LLM 回答的截图/录屏**。

**方案 B**: 在 `.env.example` 旁增加一个**一键测试脚本** `test_llm_mode.py`，模拟一个有 Key 的场景——调用一次 LLM API 后立即将结果写入缓存，之后 Fallback 优先读缓存。但这不是真实的 LLM Agent。

**方案 C**: 本地小模型（如 Qwen2.5-0.5B Instruct via llama.cpp），但依赖重、启动慢、效果差。

**我的判断**: 选 **方案 A**。理由：
- 不增加代码复杂度
- 录屏/Screenshots 比代码模拟更直观
- 面试官自己配 Key 跑一遍才是最有说服力的测试
- 那些 "优秀的面试方案" 也不需要自带 API Key

**具体行动**:
1. README 加一段 "LLM 模式 vs Fallback 模式对比"（附带示例回答对比）
2. 录一段 LLM 模式的浏览器操作视频（1 分钟）或至少截 3 张图
3. `.env.example` 已经清晰，不用改

#### P1-2: Phase 2 snippet 质量修复（15 分钟）

**文件**: [vector_store.py:96](backend/phase2/vector_store.py#L96)

**问题**: V2 搜索结果中 `snippet` 来自 ChromaDB chunk 的前 200 字符，是随意截断的，不是与查询最相关的片段。

```python
# 当前（vector_store.py:96）
"snippet": (results["documents"][0][i] or "")[:200],
```

**改法**: 在 `hybrid_search` 方法中，对 RRF 排序后的结果，从 `documents` 表查询并生成与查询相关的 snippet（复用 Phase 1 的 snippet 生成逻辑）。

或者更简单：chunk 文本本身就是从段落切出来的，平均长度 ~200-300 字符，直接返回完整 chunk 文本即可（不加截断）。

```python
# 改为返回完整 chunk 文本
"snippet": (results["documents"][0][i] or ""),
```

**影响**: 面试官看到的结果摘要更有意义，不是"随机截断的前 200 字"。

#### P1-3: 启动速度优化（10 分钟）

**问题**: 冷启动约 30-45 秒（模型加载 ~20s + ChromaDB 索引 ~10s）。对于 demo 场景偏慢。

**分析**: 主要瓶颈是 sentence-transformers 首次加载模型（~120MB 下载或本地加载）。如果模型已缓存，加载约 5-10 秒。

**改法**: 
1. `embeddings.py` 的 `get_model()` 增加预加载提示（logging）
2. 不改变架构，但可以在服务器启动日志中显式展示加载进度

```python
# embeddings.py get_model() 加一行日志
import logging
logging.info("Loading embedding model (this may take a few seconds on first run)...")
```

**不做**: 模型量化（太复杂）、预加载到内存（太重）、ONNX 导出（不可控）

## 三、Q1 深入分析：API Key 配置策略

### 题目对独立运行的要求

回顾 `question-1/README.md` 的要求：
- "构建一个 On-Call 助手 Web 应用"
- "需要实现 HTTP API + 前端页面"
- 编程语言不限，可以使用任何 AI 工具辅助
- Phase 3 要求 Agent 通过对话回答用户问题

**关键判断**: 题目要求构建的是**一个可以独立运行的 Web 应用**，不是 Claude Code 里的一个脚本。面试官需要能够启动服务器、访问页面、测试功能。

### 当前状态

| 组件 | 是否需要外部依赖 | 状态 |
|------|-----------------|------|
| Phase 1 (FTS5) | 无 | ✅ 完全独立 |
| Phase 2 (ChromaDB + MiniLM) | 首次需下载模型 120MB | ✅ 模型缓存后可离线 |
| Phase 3 (Fallback) | 无 | ✅ 完全独立 |
| Phase 3 (LLM) | 需要 LLM API Key | ⚠️ 需用户自行配置 |
| 前端 (Tailwind + Alpine.js CDN) | 需网络加载 CSS/JS | ⚠️ 首次需网络 |

### 结论

**不需要把 API Key 内嵌到项目里。** 理由：

1. 题目不要求提供 LLM Key，Phase 3 的 Fallback 模式展示了完整的 Agent 流程
2. 面试官如果有自己的 Key（DeepSeek/OpenAI），配置 `.env` 只需 30 秒
3. 硬编码 Key 是不专业的做法（安全红线）
4. 业界优秀方案都是 "自带 Key" 模式

**但需要做的是**：在 README 显眼位置告诉面试官：
- Fallback 模式能看到什么（工具调用、SSE 流、文档读取）
- LLM 模式能看到什么（完整的推理 + 综合回答）
- 怎么 30 秒配置 Key 切换到 LLM 模式

## 四、Q2 深入分析：Phase 2/3 的质量是否满足面试期望

### Phase 2 语义搜索

**题目要求**:
> "实现语义搜索，查询词不需要在文档中精确出现"

**当前达成度**: 100% — V2-F4 已证明"容器编排平台故障"（八字全不在任何 SOP 中）返回了 7 个相关结果。

**面试官可能追问的点**:
1. "为什么用 RRF 而不是加权求和？" → README 已解释（无需调参、分数分布不敏感）
2. "MiniLM 384 维对中文够吗？" → 够用，同语言匹配精度高；跨语言（中英）略有损耗但题目查询都是中文
3. "为什么不用重排模型（re-ranker）？" → 100 份文档不需要，RRF 已经够；可以提一下"如果有 1000+ 文档会加 BGE-Reranker"

**结论**: Phase 2 质量满足面试期望。不需要改动核心算法。

### Phase 3 Agent

**题目要求**:
> "实现一个 Agent，通过对话回答用户的 On-Call 问题"
> "对话过程展示 Agent 的工具调用过程"

**当前达成度（Fallback 模式）**: 工具调用流程 100% 展示，回答质量受限于"无 LLM 推理"。

**Fallback 模式的回答长什么样**:
```
正在检索相关 SOP 文档...
找到 3 个相关文档，正在读取...
## 后端服务 On-Call SOP (sop-001)
[1000 字文档内容]
---
## SRE基础设施 On-Call SOP (sop-004)
[1000 字文档内容]
---
## 移动客户端 On-Call SOP (sop-007)
[1000 字文档内容]
---
建议: 请根据以上 SOP 文档内容进行故障排查。
如需更精确的 AI 分析，请配置 OPENAI_API_KEY 环境变量。
```

**面试官视角**: 这展示了 Agent 的**检索能力**，但没有展示**推理能力**。对于 40 分的 Phase 3，只看 Fallback 模式会给面试官留下 "Phase 3 只是 Phase 2 + 自动读文件" 的印象。

**如果有 LLM 模式**:
```
Agent 正在分析问题...

[读取 sop-001]
[读取 sop-004]
[读取 sop-007]

根据三份 SOP 的综合分析：
1. 服务 OOM 的可能原因：堆内存溢出（sop-001 场景二）、...
2. 排查步骤：先检查 JVM 参数（sop-001）→ 确认 K8s 资源限制（sop-004）→ ...
3. 临时缓解：重启服务 + 扩容（sop-001 降级策略）
4. 升级路径：如持续 OOM，联系后端主管（sop-001 升级规则）
```

**这就是 LLM 模式的价值——多文档综合推理。** Fallback 模式做不到这个。

### 怎么办？

**如果我有 LLM API Key（DeepSeek/OpenAI）**: 
- 配置 `.env` → 切换到 LLM 模式 → 跑一遍 Phase 3 测试 → 截屏/录屏 → 放到 README

**如果没有**:
- 方案1: DeepSeek API 极便宜（￥1/百万 token），充 10 块够跑几百次测试
- 方案2: 用 Ollama 跑本地模型（如 Qwen2.5-7B，但需要 GPU）
- 方案3: 保持 Fallback，在 README 中详细描述 LLM 模式的预期效果

**我的建议**: 如果你有 DeepSeek Key，配上去跑一次 LLM 模式的完整测试，录屏 1 分钟，把效果写进 README。这会让 Phase 3 的 40 分拿得更稳。

## 六、不改的东西（及理由）
